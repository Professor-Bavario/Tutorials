# Arrays, Pointers and Strings

So far we have only seen singular variables into which we can store exactly one value of a given type. But what if we want to store more into a single variable?

## Arrays
This is where arrays come in. Arrays in C allows you to store multiple values into one variable, for example:
```c
int numbers[3] = {1, 2, 3};
```
Every entry has its own unique index that allows you to retrieve the value, starting from 0:
```c
numbers[0]; // == 1
numbers[1]; // == 2
numbers[2]; // == 3
```
In memory, all these values are stored right next to each other.
### Sizes
In C, Arrays have a **fixed** size. They cannot ever grow or shrink once that size has been determined. *(In the future we will define our own type that actually can do that!)*

You can specify a size for an array in the square brackets, which is required if you only declare the array:
```c
float numbers[4];
```
If you initialize the array though, you can also leave out the size and it will be determined automatically:
```c
bool checks[] = {true, false}; // checks will have 2 values
```
To get the amount of values in any given array, you can do the following:
```c
sizeof(checks) / sizeof(bool);
// or
sizeof(checks) / sizeof(checks[0]);
```
This gets the whole size in bytes of the entire array and divides it by either the size of the type or the size of the first entry (both are equal, so it does not matter which one you pick).

### Reading/writing out of bounds
This raises an important question: What if you accidentally read from outside an array, like this?
```c
int oops[2] = {4, 6};
oops[2]; // reading out of bounds!
```
In C, this is **undefined behavior**. That means you're in the wild west of programming where there are no clear laws anymore. Everything is an illusion 👻...

No but seriously, **undefined behavior** means that it's **unclear what will happen**. The code will compile and run, but we don't know what will happen.

In some cases, you might read garbage data you were not meant to read. Sometimes your program might crash from a *Segmentation Fault* (more on this in the future).

Out of bounds writing could potentially be even worse since you might corrupt important memory.

*The good news: You can only really corrupt some parts of your process, because if this happens with data you are not meant to access, a segfault will trigger and crash the program. That can still be problematic though!*

So in short: **Never read (or write!) into out of bounds!**

## Pointers
Here's a secret: There is actually no special "Array" data type of sorts. Once you compile your program, your array variable will actually be turned into a **pointer** that points to the start of your array.

A pointer **points** to a specific location in memory. Let's say we have the number 3 stored in memory at address 0x80004000:
```
0x80004000: 4
0x80004004: <some other stuff>
...
```
A pointer's value will now be that exact address:
```c
int four = 4;                   // == 4
int *pointer_to_four = &four;   // == 0x80004000 (here)
```
With pointers, you only have to store values in memory once, and can then simply pass around the pointer to those values to other functions so they can read and potentially modify them.

A pointer is noted down with a `*` anyhwere in between type and variable name:
```c
int* ptr;
int *ptr;
int * ptr;
```
To get a pointer from an existing variable, we use the `&` symbol in front of said variable, as we did above.

To read/write the value from/to this pointer, we need to **dereference** it:
```c
*pointer_to_four;       // Reads value 4
*pointer_to_four = 3;   // Writes value 3 to the pointer
pointer_to_four = 3;    // Changes the pointer from 0x80004000 to 3!!
```
**If an invalid pointer is dereferenced, this is - similar to out of bounds array access - undefined behavior.** You are reading from places you shouldn't read from or potentially even writing to them!

## Heap vs Stack
On your computer, there are two places in memory where data is kept, the **Heap** and the **Stack**.

These usually share the same part of memory, with the Heap growing from the bottom up and the Stack growing from the top down.

*(Side note: If these two ever collide, we have run out of memory and we will usually either get a Stack or a Heap Overflow error, causing the program to crash.)*

### Stack
The Stack is where local variables are in the current scope. Once you leave a scope, the variables declared in it are deleted.

All of these will be placed onto the Stack and can therefore no longer be used outside of `someFunc`:
```c
int *someFunc () {
    int val = 3;
    float number = 5.5f;
    int numbers[] = {1, 2, 3};
    return &val;
}
```
This means that if we return a pointer to `val` here, it will point to a value that has since been **removed from the stack**.

This is undefined behavior, because you are reading from memory that you're no longer supposed to read from.

So how do we make sure that val lasts even outside of the scope?

### Heap
Well, what we can do instead is to allocate the variable onto the Heap instead. The Heap does not delete anything when leaving the scope, so this will be our method of choice.

To do this we have to first allocate space on the Heap using `malloc` from the `<stdlib.h>` library, and then write our value to the pointer:
```c
#include <stdlib.h>
int *someFunc () {
    int *val = malloc(4);
    *val = 3;
    return val;
}
```
(`4` is the amount of bytes we want to allocate. If you're unsure of how big your type is and how much you need to allocate, you can use `sizeof(<type>)` as well.)

This works, but now we have a new issue:

### Memory Leaks
If we allocate something on the Heap, it never gets removed ("freed"). This means that even when our program exits, the data still exists.

This leads to **Memory Leaks**, where data can leak outside of our program and be used by other processes. On top of this, we never free this part of the Heap, which means that if we leak more and more memory, we will eventually run out of Heap space and our program will crash.

*(Memory leaks are a problem in Pokemon Scarlet & Violet for example: Data allocated on the Heap is never freed, which over time leads to glitches and crashes in-game.)*

To ensure our memory doesn't leak, we need to always free our allocated space, like this:
```c
#include <stdlib.h>
int *someFunc () {
    int *val = malloc(4);
    *val = 3;
    return val;
}
// Later, after using val for the last time...
free(val);
```
**Using a pointer again after it's freed is undefined behavior, so only free it when you do not need it anymore!**

## Strings
There is no dedicated string type in C. Instead it is represented as a `char` array or a `char` pointer:
```c
char[] str1 = "Hello guys!";
char *str2 = malloc(10);
*str2 = "I like C!";
```
Now, C needs to know when a string ends so that it doesn't read garbage data after the string.

Therefore, at the end of every string, there is a so called "NULL Terminator" character. Let's look at the string `"Hello!"` in memory (again using address `0x80004000` as an example location):
```
0x80004000: 0x48    // 'H'
0x80004001: 0x65    // 'e'
0x80004002: 0x6C    // 'l'
0x80004003: 0x6C    // 'l'
0x80004004: 0x6F    // 'o'
0x80004005: 0x20    // '!'
0x80004006: 0x00    // '\0'
```
As you can see, at the very end of the string, there is an additional character that just tells C that the string ends here. This is the NULL Terminator.

**You need to account for this extra char when you allocate memory for a string!**
```c
char *test = malloc(3);
*test = "Hey";
// Not enough! There is no space for the NULL Terminator!
```
This means you always need to allocate the length of the string + 1 bytes.
Same thing goes for `char` arrays:
```c
char test[3] = "Hey";
// Not enough! There is no space for the NULL Terminator!
```

## Pointers are just numbers
Because pointers are just addresses, we can do anything with them that we can do with normal numbers, such as *addition*. This is an alternative way to get a value from an array:
```c
int ptr[] = {1, 2, 3};

ptr[1];
// is equivalent to
*(ptr + 1 * sizeof(int));
// is equivalent to
*(ptr + 1 * sizeof(*ptr));
// is equivalent to
*(ptr + 1 * 4);
// is equivalent to
*(ptr + 4);
```
