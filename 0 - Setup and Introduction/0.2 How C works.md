# How does C work?

C and C++ are so-called **compiled** languages. This means that code you write in it will go through these three stages:

![compile journey](assets/compile%20journey.png)

Your own written C code will first be **compiled** into Assembly code. This is done by using a **compiler** like GCC, Clang, MSVC, ...

Afterwards this assembly code gets assembled (turned into binary).

For example, let's take a look at this snippet:

```c
printf("Hello World!\n");
return 0;
```
After compiling this snippet with GCC, we get Assembly code that looks something like this on an x86 machine:

![ASM example](assets/asm%20example.png)

These instructions then get assembled into a binary, that the processor can execute. This is usually done without you ever noticing, unless you specify the `-S` flag when compiling which will output the generated Assembly into a seperate `.s` file.

There are multiple different Assembly languages which are *tied to your architecture*. The computer you are currently using probably runs on the x86 architecture, whose Assembly language is very different to other Assembly languages like ARM, RISC-V, ...

*This means that C code is **architecture-specific**. Once it's compiled, it can only be used on the same architecture (and the same operating system) as it was compiled for.*

Fun Fact: The Nintendo GameCube and Wii use IBM's **PowerPC** architecture.

## Other systems
C and C++'s system is rather simple, but has some limitations. For example, you always have to compile code and this has to be done seperately for every OS and architecture you want to target.

Other languages work in different ways to try and fix some of these limitations, and I will briefly go over a couple.

### Python and JavaScript
Python and JavaScript are **interpreted** languages. This means that, instead of compiling a program *before* it runs, the code gets read *while* the program runs.

This has the benefit of you being able to use the same code on all kinds of OSes and architecture, plus you can run code on the fly instead of having to compile it first always.

On the other hand, this massively slows down performance, which means these languages are usually not viable for high-performance applications (like games).

### Java and C#
Java and C# have a very interesting system to try and have both benefits of compiled and interpreted languages.

The code you write is first compiled to **bytecode**, which is a special form of machine code (Java: JVM, C#: IL). This machine code is then run on **a virtual machine** when the program runs.

Because this virtual machine always works the same no matter your architecture or operating system, this means the same code can be used across different systems, while at the same time being relatively efficient compared to other interpreted languages.
